# Research — PWA Install & App Shell (002)

Phase 0 decisions. Each item: **Decision / Rationale / Alternatives considered**. Unknowns from the spec and Technical Context are resolved here; none remain marked NEEDS CLARIFICATION.

---

## R1 — PWA generation tooling

- **Decision**: Use `vite-plugin-pwa` with the **`generateSW`** strategy (Workbox under the hood) and `registerType: 'autoUpdate'`. The plugin emits the service worker and the web app manifest, and injects registration.
- **Rationale**: It is the canonical PWA integration for the Vite/SvelteKit toolchain already in use (001). `generateSW` precaches the build manifest (the app shell) with no hand-written Workbox code, which is exactly the shell-only need here. `autoUpdate` keeps the installed shell current without a custom update-prompt UI (YAGNI — there is no app content to lose on refresh yet).
- **Alternatives considered**:
  - Hand-written service worker + hand-authored `manifest.webmanifest`: rejected — more code, more ways to get precache/versioning wrong, violates §V.
  - `injectManifest` (custom SW with Workbox precache injected): rejected for now — needed only when you write custom runtime caching/routing; there is no API to cache yet (that arrives with 004/005). Revisit then.
  - SvelteKit's built-in `service-worker.js` hook: rejected — gives a raw SW with no Workbox precache manifest; we would reimplement what the plugin provides.

## R2 — Icons & iOS splash from a single SVG

- **Decision**: Use `@vite-pwa/assets-generator` driven by `pwa-assets.config.ts`, with the source `frontend/static/icons/icon.svg` (fetched from the provided URL). Generate the standard maskable/any PWA icon set plus the Apple touch icon and the iOS launch ("splash") image set; reference them from the manifest and `app.html`.
- **Rationale**: Single source of truth (one SVG) → all sizes generated deterministically at build time, satisfying FR-001/FR-017 and SC-009 without hand-cropping. The generator has a built-in preset covering the icon sizes browsers require for installability and the Apple splash sizes for current iPhones/iPads.
- **Alternatives considered**:
  - Commit pre-rendered PNGs: rejected — drifts from the source, bloats the repo, must be regenerated by hand on any icon change.
  - Skip iOS splash images: rejected — the spec explicitly requires splash screens; without them an installed iOS PWA shows a blank white launch.
- **Note**: iOS splash `<link rel="apple-touch-startup-image">` tags are media-query-specific; the generator emits the tag set to paste/inject into `app.html`.

## R3 — iOS detection & version parsing

- **Decision**: Derive a **Device Capability Profile** on the client from `navigator.userAgent` + feature checks. iOS Safari ≈ UA contains `iPhone|iPad|iPod` (or iPadOS desktop-mode: `Macintosh` + `maxTouchPoints > 1`) and is not another in-app browser. Parse the iOS version from the `OS xx_yy` token; compare against `16.4`. Treat **push-capable** = (not iOS) OR (iOS ≥ 16.4). If the version is unparseable, assume the degraded case (show the warning) per the spec edge case.
- **Rationale**: There is no reliable JS API for "is this iOS Safari" or "what iOS version"; UA sniffing is the accepted approach and is the only way to detect the 16.4 push boundary. Failing safe (assume degraded) avoids promising push that will not arrive.
- **Alternatives considered**:
  - Pure feature detection (e.g., presence of `Notification`/`PushManager`): rejected as sole signal — iOS exposes these only when installed+standalone, so a pre-install Safari tab cannot be distinguished from "unsupported" by features alone; the version warning must show *before* install.
  - `navigator.userAgentData`: rejected as primary — not available in Safari.
- **Fragility acknowledged**: UA strings change; the parser is isolated in `capability.ts` with unit tests pinning the 16.4 boundary and the iPadOS desktop-mode case, so future UA drift is a localized fix.

## R4 — Install-first gating

- **Decision**: Gate on **display mode**. `window.matchMedia('(display-mode: standalone)').matches` (plus iOS `navigator.standalone === true`) means "installed/launched from home screen" → render the real app shell. Otherwise render only the coached A2HS flow. Capture `beforeinstallprompt` (Chromium) to drive a native install button; on iOS Safari (no such event) render the manual Share → "Add to Home Screen" steps.
- **Rationale**: Display-mode is the standard, reliable signal for "running as an installed app" and is exactly the install-first gate the clarification requires (FR-016, SC-008). `beforeinstallprompt` is the only way to trigger the native Android/desktop-Chromium install UI.
- **Alternatives considered**:
  - Gate on a localStorage "installed" flag: rejected — unreliable (cleared storage, multiple devices) and spoofable; display-mode reflects reality.
  - Allow in-browser usage with a banner: rejected — contradicts the install-first clarification (2026-05-28).

## R5 — Notification permission timing & platform gating

- **Decision**: Request `Notification.requestPermission()` only **after** the app is in standalone mode AND only on **push-capable** platforms (per R3). On iOS <16.4, never call it. Record the outcome in Onboarding State so it is never re-prompted. Actual push subscription (`PushManager.subscribe` / VAPID) is **out of scope** → feature 005.
- **Rationale**: Requesting a permission the platform cannot fulfill (iOS <16.4) is user-hostile and pointless; gating to push-capable platforms after install matches the clarified FR-006. Capturing the grant early (in 002) lets 005 subscribe without re-prompting.
- **Alternatives considered**:
  - Request on first interaction regardless of platform: rejected — premature prompt before install hurts grant rates and violates FR-006.
  - Defer the permission request entirely to 005: considered; the clarification chose "request after install on push-capable platforms," so 002 owns the timing.

## R6 — Persistent storage

- **Decision**: Call `navigator.storage.persist()` once on first launch (guarded by Onboarding State), log the granted/denied outcome, and continue regardless. Feature-detect (`navigator.storage?.persist`) and no-op where unsupported.
- **Rationale**: Protects the precached shell and future keys/messages from eviction (FR-007). Best-effort by design — browsers may grant silently or based on engagement; the app must not depend on the result (SC-006).
- **Alternatives considered**: Gating app start on a granted result — rejected (the API is advisory; many browsers deny or defer).

## R7 — Testing strategy

- **Decision**: Two layers.
  1. **Playwright e2e** (`frontend/e2e/`) with two projects: a **Chromium** project (manifest reachable at `/manifest.webmanifest`; service worker registers and `navigator.serviceWorker.controller` is non-null after reload; `beforeinstallprompt` path renders the native install affordance) and a **WebKit** project with an injected iOS user agent + `standalone=false` (the iOS coached A2HS steps render; iOS <16.4 UA renders the version gate). Playwright's `webServer` runs `vite preview` over the production build.
  2. **Go integration test** extending `backend/test/integration/` (reusing the 001 testcontainers-Postgres harness that boots the real binary): assert the binary serves `/manifest.webmanifest` with `Content-Type: application/manifest+json` and the service-worker script with a JavaScript content type — the true production serving path through `embed.FS`.
  Plus **Vitest** unit tests for `capability.ts` (UA/version parsing).
- **Rationale**: Playwright in a real engine is the only way to assert service-worker control and the install/coaching UI; WebKit + iOS-UA is the closest automatable approximation of iOS Safari (true on-device Safari is manual). The Go test guarantees the *serving* contract that Playwright-against-`vite preview` cannot (Vite's dev content types differ from the Go file server). Splitting by concern keeps each tool authoritative for its boundary (§II).
- **CI/pre-commit**: Playwright runs in **CI only** (a new `ci.yml` step: install browsers, build, `vite preview`, run e2e) — it is too heavy for the pre-commit hook (§C unchanged). The Go serving test runs in the existing `integration` job.
- **Alternatives considered**:
  - Playwright against the Go binary for everything: rejected as the default — couples every frontend assertion to a Go build + Postgres boot; instead the Go test owns only the serving contract.
  - "Real iOS Safari driver" (BrowserStack/Sauce/simulator): noted as out-of-scope for automated CI (cost/flake); WebKit emulation covers the rendering-path assertions, real-device verification is a manual quickstart step.

## R8 — Serving correctness from Go `embed.FS`

- **Decision**: Register `mime.AddExtensionType(".webmanifest", "application/manifest+json")` in `internal/web` before constructing the file server. Serve the service worker from the build root (e.g. `/sw.js`) so its default scope is `/` — no `Service-Worker-Allowed` header needed. Rely on Go's built-in `.js` → JavaScript MIME mapping for the SW. Service workers require a **secure context**: dev is already HTTPS via Caddy's internal CA (`ring.localtest.me`), and `localhost` is treated as secure, so registration works in both dev and prod.
- **Rationale**: Go's `mime` package does not know `.webmanifest` by default (it would fall back to `application/octet-stream`, which some browsers reject for a manifest). The one-line registration is the minimal serving fix and stays within "serving the new static assets" (no behavioral backend change). Root-scoped SW avoids extra headers.
- **Alternatives considered**:
  - A dedicated `GET /manifest.webmanifest` handler: rejected — unnecessary; the file server already serves it once the MIME type is known.
  - Custom header middleware for the SW: rejected — not needed at root scope.

## R9 — Service worker × `adapter-static` / SvelteKit

- **Decision**: Keep `adapter-static` with `prerender = true` (from 001). Configure `vite-plugin-pwa` so its precache `globPatterns` cover the prerendered HTML + `_app/` immutable assets. Service-worker registration is invoked from the root layout on first load (the plugin's virtual `registerSW` module).
- **Rationale**: The shell is fully static/prerendered, so Workbox precache of the build output is exact and cache-busted by Vite's content hashes. Registering from the layout ensures it happens on first page load (FR-002).
- **Alternatives considered**: SvelteKit `@sveltejs/adapter-static` + manual SW registration in `app.html` — rejected; the plugin's `registerSW` handles lifecycle (incl. `autoUpdate`) correctly.

## R10 — Offline behavior & precache scope

- **Decision**: Precache the app shell (prerendered routes + immutable assets + icons). Because the app is install-first, the **coaching view is part of the shell**, so a returning pre-install visitor still gets a coherent screen offline. First-ever visit with no network cannot precache → show a clear loading/error state, not a blank page (spec edge case). No runtime API caching (none exists yet).
- **Rationale**: Satisfies the §Offline-tolerance constraint and SC-002 (≤2s shell render offline) with the smallest precache surface. Defers runtime caching strategy to when there is actual dynamic content (004/005).
- **Alternatives considered**: Aggressive runtime caching / background sync — rejected now (YAGNI; no API).
