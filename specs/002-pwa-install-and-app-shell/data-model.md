# Data Model — PWA Install & App Shell (002)

This feature introduces **no server-side data and no database tables**. All state is client-side. Two runtime entities drive behavior, and one build artifact (the manifest) has a fixed shape. The "store" is the browser: `localStorage` for durable onboarding state, derived in-memory state for capability, and Cache Storage (Workbox-managed) for the precached shell.

---

## 1. Device Capability Profile (derived, in-memory)

Computed once per page load from `navigator` + media queries. Never persisted (it is a function of the current environment). Drives which view renders and whether to request notification permission.

| Field | Type | Source / Rule |
|---|---|---|
| `platform` | `'ios' \| 'android' \| 'other'` | UA classification |
| `isIOSSafari` | `boolean` | iOS UA (incl. iPadOS desktop-mode: `Macintosh` + `maxTouchPoints > 1`) and not a known in-app/embedded browser |
| `iosVersion` | `{ major: number, minor: number } \| null` | parsed from the `OS xx_yy` UA token; `null` if unparseable |
| `isStandalone` | `boolean` | `matchMedia('(display-mode: standalone)').matches \|\| navigator.standalone === true` |
| `installPromptAvailable` | `boolean` | a `beforeinstallprompt` event has been captured and not yet consumed |
| `pushCapable` | `boolean` | `platform !== 'ios'` OR (`iosVersion.major > 16` OR (`major === 16` && `minor >= 4`)). If `isIOSSafari && iosVersion === null` → `false` (fail safe) |

**Derivation rules**:
- `pushCapable` deliberately ignores `isStandalone` for the *warning* decision (iOS <16.4 must be warned even pre-install), but the notification-permission request additionally requires `isStandalone` (see Onboarding State transitions).
- Unparseable iOS version ⇒ treat as not push-capable (show the degradation warning) — never promise push that may not work.

## 2. Onboarding State (persisted: `localStorage`)

Durable, per-browser-profile record so the app does not re-prompt for already-decided things and can track first-launch one-time actions. Single key (e.g. `ring.onboarding.v1`) holding a JSON object.

| Field | Type | Default | Meaning |
|---|---|---|---|
| `schemaVersion` | `number` | `1` | migration guard for future shape changes |
| `firstLaunchAt` | ISO 8601 string | set once | timestamp of first launch (guards one-time actions) |
| `persistRequested` | `boolean` | `false` | `navigator.storage.persist()` has been attempted |
| `persistGranted` | `boolean \| null` | `null` | outcome of the persist request (`null` = not yet / unsupported) |
| `notificationPermission` | `'default' \| 'granted' \| 'denied' \| 'skipped'` | `'default'` | `'skipped'` = intentionally not requested (push-incapable platform); otherwise mirrors the browser permission decision |

**Validation / invariants**:
- `notificationPermission` is only moved off `'default'` *after* `isStandalone` is true AND `pushCapable` is true (else it stays `'default'`, or is set to `'skipped'` on a push-incapable platform).
- Once `notificationPermission ∈ {granted, denied}` the app never calls `requestPermission()` again.
- `persistRequested` flips to `true` exactly once (first launch); the request is not retried.

**State transitions** (per launch):

```
launch
  ├─ if !persistRequested: call storage.persist() → set persistRequested=true, persistGranted=<result>   [log storage.persist]
  ├─ compute Device Capability Profile
  ├─ if !isStandalone:
  │     ├─ render coached A2HS flow (InstallCoach)                                                        [log coach.shown]
  │     └─ if isIOSSafari && !pushCapable: render IOSVersionGate warning                                  [log versiongate.shown]
  └─ if isStandalone:
        ├─ render app shell                                                                               [log install.completed (first standalone launch)]
        └─ onboarding-final step:
              ├─ if pushCapable && notificationPermission == 'default':
              │       call Notification.requestPermission() → store result                                [log notif.permission granted|denied]
              └─ else if !pushCapable && notificationPermission == 'default':
                      set notificationPermission='skipped'                                                [log notif.permission skipped]
```

(Bracketed items are the structured client-log events from plan §III. Actual push *subscription* is feature 005.)

## 3. Web App Manifest (build artifact, generated)

Generated by `vite-plugin-pwa`, served at `/manifest.webmanifest`. Not "data" in the runtime sense, but its shape is a contract (see `contracts/manifest.schema.json`). Required shape:

| Field | Value / Rule |
|---|---|
| `name` | `"Ring"` |
| `short_name` | `"Ring"` |
| `start_url` | `"/"` |
| `scope` | `"/"` |
| `display` | `"standalone"` (required for the install-first standalone launch) |
| `theme_color` / `background_color` | brand colors (also used for the iOS splash background) |
| `icons[]` | generated from `icon.svg`; MUST include the install sizes and at least one `purpose: "maskable"` icon |

## 4. Onboarding State migration note

`schemaVersion` exists so later features (e.g. 005 storing a push-subscription marker) can extend the object without clobbering a returning user's prior state. On load, if the stored `schemaVersion` is unknown/newer, the app resets to defaults (safe — onboarding re-runs are cheap; the only cost is a possible re-request of an undecided permission). This keeps the client contract forward-compatible per Principle IV.
